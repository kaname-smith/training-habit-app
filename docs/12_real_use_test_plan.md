# 12 Real Use Test Plan — 実利用テスト計画(v0.2.0-phase2)

## 1. 位置づけ

`v0.2.0-phase2` タグの時点で、Phase 1(MVP)とPhase 2(GUI/UX強化・文言一元化・疑似通知・E2E・PWA manifest軽微修正)が完了している。Phase 3(Service Worker、PWAアイコン、ダークモード作り込み)に進む前に、実際の試験期間中の生活の中でこのアプリを使い、機能テストでは見えない使用感・不具合・改善点を集める。

## 2. 実利用テストの目的

- 型チェック・lint・自動テストでは検出できない「実際に毎日使ったときの違和感」を洗い出す。
- 7月試験期間という本来の利用文脈(疲労・時間のなさ・夜間利用)の中で、習慣化ロジックと文言が意図通り機能するかを確認する。
- Phase 3で本当に必要な機能(Service Worker、通知、ダークモード作り込みなど)の優先順位を、実体験に基づいて判断できるようにする。
- 見つかった不具合・改善要望を記録し、Phase 3着手前に対応要否を判断できる状態にする。

## 3. 7月試験期間中に確認すべき観点

- 試験前日・試験当日など、疲労や時間のなさが実際に発生する日に、アプリの推奨が無理を強いていないか。
- 睡眠不足や体調不良の日に、10分版・休息への切り替えが心理的に選びやすいか。
- 毎日開いても圧迫感がなく、むしろ「開きたくなる」状態になっているか。
- 7/7〜7/31の期間中、記録が途切れても再開しやすいと感じられるか(責める文言が実際の画面に出てこないか)。

## 4. 毎日確認する項目

- [ ] アプリを開いてから、今日やることが3秒以内に分かったか。
- [ ] 今日のおすすめ(A/B/10分版/休息)が、その日の実際の体調・予定に照らして妥当だったか。
- [ ] 疲労チェックの入力が煩わしくなかったか(展開・入力の手間)。
- [ ] リマインダーバナーが出た場合、内容・タイミングが不快でなかったか。
- [ ] 当日の記録(完了・休息)がダッシュボードに正しく反映されたか。

## 5. トレーニング実行時に確認する項目

- [ ] 種目名・回数/秒数・フォーム注意が運動中に読みやすかったか。
- [ ] セットのチェック操作が親指操作で迷わず行えたか。
- [ ] 進捗ドットが「あと何種目か」の目安として機能したか。
- [ ] 「軽量版へ変更」「中止して休息記録」が、実際に必要な場面で迷わず押せたか。
- [ ] 完了画面の一言・達成コメントが、押し付けがましくなく自然に感じられたか。
- [ ] 体感強度の入力後、プロテイン記録への導線がスムーズだったか。

## 6. プロテイン/栄養記録で確認する項目

- [ ] 今日のタンパク質目標・推定摂取量が体感と一致しているか。
- [ ] プロテイン杯数ボタン(0/1/2)で実際の摂取パターンをカバーできているか。
- [ ] 食事タンパク質のステッパー入力(+/-)が、実際の入力量に対して使いやすい単位・範囲か。
- [ ] 体重別目標の説明が、試験期間の実感(不足しがち/摂りすぎ)と合っているか。

## 7. 疲労が重い日の導線確認

- [ ] 睡眠「かなり少ない」または疲労「重い」を選んだ際、推奨が休息または10分版に切り替わることを確認する。
- [ ] 「明日試験がある」をONにした際、10分版が推奨されることを確認する。
- [ ] 「痛み・体調不良がある」をONにした際、休息が推奨されることを確認する。
- [ ] これらの状態でも、文言が責める・急かす印象を与えていないかを確認する。
- [ ] 疲労が重い状態から回復した翌日以降、通常メニューへの復帰が自然に感じられるか。

## 8. 記録・カレンダー画面の見やすさ確認

- [ ] 月間カレンダーで、完了/10分版完了/休息/未記録が凡例なしでも直感的に区別できるか。
- [ ] 日をタップした際の詳細表示(実施内容・体感強度)が有用か。
- [ ] 週ごとの実施数・総実施回数が、モチベーションになるが焦りは生まないか。
- [ ] プランク最長・腕立て最多などの種目別記録が、進歩の実感として機能しているか。
- [ ] スマホの実機・実際の指のサイズでカレンダーの日付タップがしやすいか。

## 9. 設定・リセット・JSONエクスポートの確認

- [ ] 体重・プロテイン1杯量・試験モードの変更が即座に反映されるか。
- [ ] 通知のおすすめ時刻設定と、実際のリマインダーバナー表示タイミングが一致するか。
- [ ] データエクスポート(JSON)でダウンロードされる内容が正しいか(実機のダウンロード動作含む)。
- [ ] データリセットの2段階確認が、誤操作を防ぐのに十分か。
- [ ] リセット後、初期設定画面から正しく再開できるか。

## 10. GUI/UXフィードバック記録欄

実際に使っていて感じた「良い点」「気になる点」を都度追記する。日付・画面・内容の3点を最低限記録する。

```text
| 日付 | 画面 | フィードバック内容 |
|---|---|---|
|  |  |  |
```

## 11. 不具合記録欄

再現手順が分かる不具合は、Phase 3着手前の修正候補として記録する。

```text
| 日付 | 画面/操作 | 期待した動作 | 実際の動作 | 再現手順 |
|---|---|---|---|---|
|  |  |  |  |  |
```

## 12. Phase 3に進む判断基準

以下を満たした場合、Phase 3(Service Worker・PWAアイコン・ダークモード作り込みなど)への着手を検討する。

- 数日〜1週間程度、実際の試験期間中の生活の中で継続利用できている。
- 「不具合記録欄」に、機能停止級の重大な不具合が残っていない。
- 「GUI/UXフィードバック記録欄」の内容から、Phase 3で優先すべき項目(あるいは不要な項目)が判断できる状態になっている。
- 8月以降の本格強化モードに向けて、現状のPhase 1/2の範囲で習慣化という目的が達成できていると判断できる。

上記が揃わない場合は、Phase 3ではなくPhase 1/2の範囲内の小さな修正・調整を優先する。
